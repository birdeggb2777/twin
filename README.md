# 自然語言

這個程式語言有兩個名字，分別為「以自然語言樣貌呈現的程式語言」、「披著自然語言外衣的高階語言」。

還在概念驗證階段，只有十個語法，文字也都還沒打好，且是在達克效應的山谷上打的，請不要閱讀。

--------------

**開發緣由**

在十年前，我在高中閱讀計算機概論的時候，書上提到了程式語言有四代(又有一說是五代)：
「機器語言」→「組合語言」→「高階語言」→「自然語言」

編譯器可以將高階語言翻譯成組合語言，組譯器可以將組合語言翻譯成機器語言。

不過，這個叫「自然語言」的第四代語言是什麼，就是指人類平常在說的語言，不過當時書上表示自然語言的研究還在進行。

如果在遙遠的未來真的做出來了，軟體的未來應該才真正要開始吧 o(*≧▽≦)ツ

--------------

在後來幾年，從高階語言學到組合語言，接著開始做一些不自量力的事(現在也是)。

有一本教編譯器原理的書告訴我「編譯器也只不過是文字轉文字的普通程式」，不過我忽略了這個句子後面多了個「但是...」

在嘗試實作直譯器和語法樹時，這對我來說非常困難，但從中學到遠比語法本身還重要的東西。

若真的遇到擅長寫組合語言的人，或許會忍不住肅然起敬吧！

--------------

可惜到了今天，能夠讓軟體的未來真正開始的自然語言仍尚未出現，軟體就開始被許多人喧為毫無未來的領域。

不過，最近我聽到一個說法：Github有80%的程式是由AI所產生的；但又有些人提出了一種疑問：「編譯器是否也是個AI」。

喔，看了之後我想了一下，這個「自然語言」的技術會不會其實已經悄悄實現，但從沒被我注意到，因為它被置換成了「AI」這個詞彙。

組譯器(assembler)，編譯器(compiler)，在加上這個transformer(變換器)，就可以串起來了。

但以下行數這個疑問，讓我難以確認自然語言是否真的已經被發明了。

--------------

**程式碼行數的計算**

以程式碼行數衡量進度這件事情，早已不是這一兩年才流行過的事。

在過去幾年，大概依稀知道了用「cout」和「printf」印數字，會有天壤之別。

如果只需要印數字，cout比printf更臃腫、速度更慢(但行數可以賺更多)；但無論如何，我都不曾想過編譯後的組合語言，可以當作是程式碼行數的計算。

這部分感覺好像有些矛盾，如果提示詞被認為是程式碼的一種，那為什麼計算產出時，是用變換器編譯出去的程式碼量來衡量。

所以，提示詞是不是自然語言的程式碼，以下幾個議題，讓我延伸出幾個我認為或許需要解決的問題。

--------------

**1. 重新定義程式碼行數**

當我寫了一條提示詞「幫我隨意寫個購物網站，就這樣。」，我應該只能得到「一行」的產出，而非「數百行」。

但這是我將提示詞視作程式碼為前提，這部分或許要有個定義方式。

然而，以下這一行，或許更應該要被嚴格重視。

> 當一個人的提示詞是「幫我將某個GPL授權條款的程式碼改掉，以我的名義重新發布。」，它的產出(程式碼擁有權)應該也得是這一行，不該是標竊(或import)來的程式碼。

**2. 於Github上提交「編譯後的程式碼」現象**

有些開源人會拒絕生成式模型提交的程式碼，要求對方直接提供提示詞，比較容易了解意圖。

乍看之下，提示詞似乎可以做為程式碼的交付方式，但兩個相同的提示詞，在經過各自的轉換器(transformer)翻譯後，依然會產生不同的結果。

> 高階語言有「C」、「Python」，但自然語言可沒有個「標準」。

顯然，提示詞嚴格上不能算是個自然語言。

而且，有些人並不喜歡透漏太多「意圖」。

**3. 大型模型過度集中於某個地方**

跟之前某幾個技術潮流一樣，宣稱去中心化、人人可有，但某些人總想要讓技術變質，反過來說服所有人，讓東西都進到同一個主機上統一管理。

對我而言，小型語言模型(SLM)比大型語言模型更值得研究，它只專注於特定的任務，不會弄到碳排放全面失控、也不會消耗過多無用電力、開發的目的也更不容易是一些無助於人類福扯的東西。

> 或許有人會說，SLM怎麼可能會做比大模型更好，但，這不就是我們工程師應該要做的事情嗎。

**4. 中間還有些字沒打完**

(中間還有些字沒打完)

**真正實質意義上的自然語言**

基於以上原因，我認為我在許多情況難以接受過度抽象化的程式，所以嘗試設計一個「確定性的」自然語言。

讓使用者透過提示詞產生程式磚，程式磚可以透過已知的傳統程式「直接對應」到傳統程式碼。

所以無論轉換器換成哪個，它都不會產生過於不一致的結果。

這樣做也有一個好處，那就是code review時可以直接閱讀中文。

**範例**

範例先從畫圖開始，因為我能力不足，時間不多，先用最容易的畫圖來達成轉換。

資料結構：

● 「顏色」(struct):R(byte)、G(byte)、B(byte)、A(byte)

● 「座標」(struct):「x」(double)、「y」(double)

● 「筆」(struct):「名稱」(str)、「顏色」(struct)、「粗細」(int)、「是否填充」(bool)

程式磚：

● 拿起名稱為<A>的筆。

● 替名稱為<A>的筆沾上新顏色為<A>。

● 替名稱為<A>的筆設定是否填充為<A>。

● 替名稱為<A>的筆設定粗細度為<A>。

● 用手中的筆，繪製正圓形在座標<,>，半徑為。

● 用手中的筆，繪製三角形，三個點的座標為<,>、<,>、<,>。
